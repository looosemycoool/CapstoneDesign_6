## 1. 문서 개요

### 1-1. 문서 목적

본 문서는 **학교 공지사항 크롤링 기반 하이브리드 챗봇** 서비스의 기능 요구사항, 화면 설계, 기술 명세를 정의한다. 개발 과정에서 팀 전체가 동일한 이해를 가지고 작업하도록 하는 단일 진실 원천(Single Source of Truth)으로 사용된다.

### 1-2. 범위

- **In Scope**: 프론트엔드(React/TS), 백엔드 API 인터페이스, 공지사항 크롤링 연동
- **Out of Scope**: 크롤러 자체 구현 세부 사항, 인증/로그인, 관리자 페이지

### 1-3. 이해관계자

| 역할 | 이름/조직 | 관심사 |
|---|---|---|
| 제품 오너 | 6조 전체 | 기능 완성도, 일정 |
| 개발 (FE) | 프론트엔드 담당 | API 스펙, 컴포넌트 설계 |
| 개발 (BE) | 백엔드 담당 | DB 스키마, 크롤링 파이프라인 |
| 최종 사용자 | 재학생 | 빠른 공지 조회, 편의성 |

## 2. 배경 및 문제 정의

### 2-1. 현황 분석

학교 포털의 공지사항 시스템은 다음과 같은 구조적 문제를 가지고 있다.

```
현재 공지사항 탐색 경로
학생 → 포털 접속 → 카테고리 선택 → 게시판 목록 → 페이지 이동 → 원문 확인
                     ↑
           (학사, 장학, 취업, 행사 등
            10개+ 카테고리로 분산)
```

| 문제 | 영향 |
|---|---|
| 카테고리가 분산되어 어떤 게시판에 공지가 있는지 모름 | 탐색 시간 증가 (평균 5분+) |
| 포털 내 검색 기능이 키워드 일치에만 의존 | 자연어 질의 불가 |
| 모바일 환경에서 포털 UX가 불편 | 접근성 저하 |
| 오래된 공지와 최신 공지가 혼재 | 정보 신뢰도 저하 |

### 2-2. 솔루션 개요

```
제안 시스템
학생 → 챗봇 입력 → [하이브리드 RAG 엔진] → 자연어 응답 + 출처 카드
                          ↑
              크롤링된 공지 DB (매일 갱신)
              + LLM 자연어 이해
```

## 6. 핵심 기능 명세

### 6-1. 채팅 인터페이스

#### F-CHAT-01: 메시지 전송

| 항목 | 내용 |
|---|---|
| 우선순위 | P0 |
| 설명 | 사용자가 텍스트를 입력하고 전송하면 챗봇에 메시지를 보낸다 |
| 트리거 | Enter 키 입력 또는 전송 버튼(↑) 클릭 |
| 전제 조건 | 입력창에 1자 이상 텍스트 존재 |
| 정상 흐름 | 1. 사용자 메시지를 우측에 즉시 표시 → 2. 타이핑 인디케이터 표시 → 3. 스트리밍 응답 수신 → 4. 응답 완료 후 출처 카드 표시 |
| 예외 흐름 | 빈 입력 시 전송 버튼 비활성화, 전송 불가 |
| 성공 기준 | 메시지가 서버에 전달되고 응답이 스트리밍으로 렌더링됨 |

**UI 상세 규칙**
- `Shift + Enter`: 줄바꿈 (메시지 전송 X)
- `Enter`: 메시지 전송
- 전송 중에는 입력창 비활성화 (스트리밍 완료 후 재활성화)
- 입력창 최소 높이: 1행, 최대 높이: 5행 (그 이상은 스크롤)

---

#### F-CHAT-02: 스트리밍 응답 렌더링

| 항목 | 내용 |
|---|---|
| 우선순위 | P0 |
| 설명 | SSE(Server-Sent Events)로 수신되는 텍스트 청크를 실시간으로 렌더링한다 |
| 기술 방식 | EventSource 또는 fetch + ReadableStream |
| 렌더링 단위 | 서버에서 보내는 `StreamChunk.content` 단위로 누적 |
| 마크다운 처리 | 스트리밍 완료 후 마크다운 최종 파싱 (중간 상태는 raw text) |
| 완료 신호 | `type: "done"` 청크 수신 시 스트리밍 종료 처리 |
| 출처 수신 | `type: "sources"` 청크에서 `AnnouncementCard` 렌더링 트리거 |

#### F-CHAT-03: 응답 중단

| 항목 | 내용 |
|---|---|
| 우선순위 | P1 |
| 설명 | 스트리밍 중 사용자가 중단 버튼을 누르면 응답 생성을 즉시 멈춘다 |
| 트리거 | "중단" 버튼 클릭 (스트리밍 중에만 표시) |
| 기술 방식 | `AbortController.abort()` |
| 중단 후 처리 | 현재까지 수신된 텍스트는 그대로 유지, 입력창 재활성화 |

#### F-CHAT-04: 멀티턴 대화

| 항목 | 내용 |
|---|---|
| 우선순위 | P0 |
| 설명 | 이전 대화 맥락을 유지하며 후속 질문에 답변한다 |
| 구현 방식 | 각 메시지 전송 시 `sessionId`를 포함하여 서버 측에서 컨텍스트 유지 |
| 세션 단위 | 한 채팅창에서의 모든 메시지 = 하나의 세션 |

#### F-CHAT-05: 마크다운 렌더링

| 항목 | 내용 |

## 9. 상태 관리 설계

### 9-1. chatStore (Zustand)

```ts
// store/chatStore.ts

interface ChatStore {
  // ─── 상태 ────────────────────────────────────────────
  sessionId: string | null;       // null = 새 대화
  messages: Message[];            // 현재 세션의 전체 메시지
  isStreaming: boolean;           // 스트리밍 진행 중 여부
  abortController: AbortController | null; // 스트림 중단 제어

  // ─── 액션 ────────────────────────────────────────────
  setSessionId: (id: string | null) => void;
  addMessage: (message: Message) => void;
  appendStreamChunk: (chunk: string) => void;  // 마지막 assistant 메시지에 청크 누적
  finalizeStream: (sources?: AnnouncementSource[]) => void; // 완료 처리 + 출처 붙이기
  setStreaming: (value: boolean) => void;
  setAbortController: (ctrl: AbortController | null) => void;
  resetChat: () => void;          // 새 대화 시작 시 상태 초기화
  loadSession: (messages: Message[], sessionId: string) => void; // 히스토리 복원
}
```

**상태 전이 다이어그램**
```
[초기] sessionId=null, messages=[], isStreaming=false
   │
   │ addMessage (user)
   ▼
[전송] messages=[userMsg], isStreaming=true
```

## 10. API 인터페이스 명세

### 10-1. POST /api/chat — 챗봇 메시지 전송

**Request**
```ts
interface ChatRequest {
  sessionId: string | null;  // null이면 새 세션 생성

```

**Response (SSE — text/event-stream)**
```
event: chunk
data: {"type":"text","content":"2학기 수강신청은"}

event: chunk
data: {"type":"text","content":" 다음과 같습니다."}

event: chunk
data: {"type":"sources","sources":[{"id":"ann_001","title":"2025학년도 2학기 수강신청 안내","category":"학사","publishedAt":"2025-06-10","url":"https://..."}]}

event: chunk
data: {"type":"session","sessionId":"sess_abc123"}

event: chunk
data: {"type":"done"}
```

**StreamChunk 타입**
```ts
type StreamChunkType = "text" | "sources" | "session" | "done" | "error";

interface StreamChunk {
  type: StreamChunkType;
  content?: string;                    // type === "text"
  sources?: AnnouncementSource[];     // type === "sources"
  sessionId?: string;                  // type === "session" (새 세션)
  error?: string;                      // type === "error"
}
```

**에러 응답 (HTTP 4xx/5xx)**
```json
{ "error": "INTERNAL_SERVER_ERROR", "message": "서버 오류가 발생했습니다." }
```

## 12. 에러 처리 시나리오

| 에러 상황 | 감지 방법 | UI 처리 | 복구 방법 |
|---|---|---|---|
| 네트워크 연결 없음 | `fetch` catch | 어시스턴트 버블에 "인터넷 연결을 확인해주세요" | 재시도 버튼 |
| 서버 500 오류 | HTTP status 500 | 어시스턴트 버블에 "서버 오류가 발생했습니다" | 재시도 버튼 |
| 스트리밍 중 연결 끊김 | ReadableStream error | 현재까지 수신 텍스트 유지 + 에러 안내 | 재시도 버튼 |
| 응답 없음 (타임아웃 30s) | AbortController timeout | 타임아웃 안내 메시지 | 재시도 버튼 |
| 히스토리 로드 실패 | HTTP error | 사이드바에 토스트 에러 | 새로고침 안내 |
| 세션 삭제 실패 | HTTP error | 토스트 에러 | 목록 유지 |
| 컴포넌트 렌더 에러 | ErrorBoundary | "화면 오류 발생" 폴백 UI | 새로고침 버튼 |

## 16. 용어 정의

| 용어 | 정의 |
|---|---|
| 세션 (Session) | 사용자의 하나의 연속된 대화 단위. 새 대화 시작 시 새 세션 생성 |
| 스트리밍 (Streaming) | SSE를 통해 응답 텍스트를 청크 단위로 실시간 전송하는 방식 |
| 청크 (Chunk) | 스트리밍 응답의 단위 데이터 |
| 출처 카드 | 챗봇 응답이 참조한 공지사항을 시각적으로 표시하는 카드 UI |
| 하이브리드 | RAG(검색 기반) + LLM 생성의 결합 방식 |
| RAG | Retrieval-Augmented Generation. 벡터 검색으로 관련 문서를 찾아 LLM에 컨텍스트로 제공 |
| 멀티턴 | 이전 대화 내용을 맥락으로 유지하며 이어지는 다회차 대화 |
| 웰컴 화면 | 대화가 없을 때 첫 진입 시 표시되는 안내 및 예시 질문 화면 |
| TTFT | Time To First Token. 메시지 전송 후 첫 번째 텍스트 청크를 수신하기까지의 시간 |
